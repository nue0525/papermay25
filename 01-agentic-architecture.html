Core Concepts

What You Must Know Cold

These concepts form the backbone of Domain 1. Every question will test your understanding of at least one of these patterns.

Concept 1

The Agentic Loop

The fundamental control flow: send request → check stop_reason → if "tool_use" execute tool and append result → loop. If "end_turn", return final response. This is THE core pattern — everything else builds on it.

Concept 2

Hub-and-Spoke Architecture

Coordinator manages ALL inter-subagent communication. Subagents have isolated context — they do NOT inherit coordinator's conversation history automatically. All routing goes through the coordinator for observability.

Concept 3

Programmatic vs Prompt Enforcement

When compliance has financial/safety consequences, use hooks and prerequisites (deterministic, 100% reliable). Prompt instructions have a non-zero failure rate and are insufficient for critical business rules.

Concept 4

Explicit Context Passing

Subagent context must be explicitly provided in the prompt. Subagents don't auto-inherit parent context or share memory. Include complete prior findings directly when spawning subagents via the Task tool.

Exam Pattern
Questions will give you scenarios where an agent misbehaves and ask you to identify the root cause or fix. The answer almost always comes down to: "use stop_reason correctly" or "use programmatic enforcement instead of prompt instructions."

Task 1.1 — Design & Implement Agentic Loops

Knowledge You Need:

K1
Agentic loop lifecycle: Send request → inspect stop_reason ("tool_use" vs "end_turn") → execute requested tools → return results for next iteration. The model drives the decision-making, not pre-configured logic.
K2
Tool result appending: Results are appended to conversation history so the model can reason about what to do next based on the data it received back.
K3
Model-driven vs pre-configured: Claude reasons about which tool to call based on context. Do NOT pre-configure tool call sequences or decision trees — let the model decide.

Skills You Need:

S1
Control flow implementation: while True → check response.stop_reason → break on "end_turn" → execute and append on "tool_use"
S2
Anti-pattern avoidance: Never parse NL to decide when to stop. Never set arbitrary max_iterations = 5 as primary control. Never check if assistant text contains "final answer" or similar markers.
Python — Correct Agentic Loop
# ✅ CORRECT: stop_reason drives the loop
messages = [{"role": "user", "content": user_query}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        system=system_prompt,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":
        break  # Model is done

    if response.stop_reason == "tool_use":
        # Find all tool use blocks
        tool_uses = [b for b in response.content if b.type == "tool_use"]

        # Append assistant message
        messages.append({"role": "assistant", "content": response.content})

        # Execute each tool and append results
        tool_results = []
        for tu in tool_uses:
            result = execute_tool(tu.name, tu.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tu.id,
                "content": result
            })

        messages.append({"role": "user", "content": tool_results})
Python — ❌ ANTI-PATTERN: Do NOT Do This
# ❌ WRONG: Arbitrary iteration cap as primary control
for i in range(5):  # Agent may need 7 steps or only 2!
    response = client.messages.create(...)
    if "I'm done" in response.content[0].text:  # ❌ NL parsing
        break

# ❌ WRONG: Checking for text content as completion
if any(b.type == "text" for b in response.content):  # ❌
    break  # Model can emit text AND tool calls in same response
Task 1.2 — Multi-Agent Coordinator-Subagent Patterns
Hub-and-Spoke Architecture — Communication Flow

Critical Knowledge:

1
Overly narrow decomposition is a key exam topic. If coordinator decomposes "AI in creative industries" into only visual arts subtopics, music/writing/film are missed. The root cause is always the coordinator's decomposition, NOT downstream agents.
2
Dynamic subagent selection: Coordinator should analyze query requirements and dynamically select which subagents to invoke — NOT always route through the full pipeline.
3
Iterative refinement loops: Coordinator evaluates synthesis output for gaps → re-delegates to search/analysis with targeted queries → re-invokes synthesis until coverage is sufficient.
Task 1.3 — Subagent Invocation, Context Passing & Spawning

The Task Tool & allowedTools

Python — Spawning Subagents with Task Tool
# Coordinator's agent definition
coordinator = AgentDefinition(
    name="research_coordinator",
    description="Coordinates multi-agent research tasks",
    system_prompt="""You are a research coordinator. Analyze the query,
decompose into subtasks, and delegate to specialized subagents.
Specify RESEARCH GOALS and QUALITY CRITERIA — not step-by-step procedures.
This enables subagent adaptability.""",
    allowed_tools=["Task"],  # ✅ MUST include "Task" to spawn subagents
)

# Subagent definitions — each has scoped tools
search_agent = AgentDefinition(
    name="web_researcher",
    description="Searches the web for information on specific topics",
    system_prompt="You search the web and return structured findings.",
    allowed_tools=["web_search", "fetch_url"],  # ✅ Scoped tools only
)

# ✅ Explicit context passing — include ALL prior findings
# Subagents do NOT auto-inherit coordinator context!
coordinator_prompt = f"""
Research topic: {topic}

The web search agent found:
{json.dumps(search_results, indent=2)}

The document analysis agent found:
{json.dumps(doc_analysis, indent=2)}

Now synthesize these findings into a comprehensive report.
Preserve source attribution (URLs, document names, page numbers).
Distinguish well-established findings from contested ones.
"""

# ✅ Parallel subagent execution — multiple Task calls in ONE response
# Coordinator emits multiple Task tool calls simultaneously
# This is faster than sequential execution across separate turns
Exam Tip
When asked about context passing between agents, the answer is almost always "explicitly include complete findings in the prompt." Auto-inheritance is a trap answer.

Task 1.5 — Agent SDK Hooks

Two Hook Patterns You Must Know

PostToolUse Hook

Intercepts tool RESULTS for transformation BEFORE the model processes them. Use for: data normalization (Unix timestamps → ISO 8601), trimming verbose output to relevant fields, format standardization.

Tool Call Interception Hook

Intercepts OUTGOING tool calls to enforce compliance. Use for: blocking operations above thresholds (refunds > $500), enforcing required prerequisites, redirecting to escalation when policy is violated.

Practice Questions

20 Questions — Domain 1

These questions cover the range of difficulty and topics you'll encounter. Study each explanation carefully — understanding WHY wrong answers are wrong is as important as knowing the correct one.

1
Agentic Loop Termination
Task 1.1 · Agentic Loops
Your agentic loop occasionally terminates prematurely, missing required tool calls. The current implementation checks if len(response.content) > 0 and response.content[0].type == "text" to decide when to stop. What's the issue?
A
The model can emit text content AND tool_use blocks in the same response. Checking for text doesn't mean tools aren't also requested.
B
The response content array is always empty when tool calls are pending.
C
The content blocks are returned in random order, so checking index 0 is unreliable.
D
Text blocks are only emitted after all tool calls have completed.
A
CORRECT
Explanation

The model can return both text and tool_use blocks in a single response. Checking for text content as a completion signal is an anti-pattern — you must use stop_reason to determine loop behavior. Only "end_turn" means the model is done.

2
Loop Control Anti-Pattern
Task 1.1 · Anti-Patterns
A team implements their agentic loop with for i in range(10): as the main loop control. The agent sometimes completes in 3 iterations but is forced to run all 10. Other times it needs 12 iterations but gets cut off. What's the correct approach?
A
Increase the cap to 20 and add early termination when the model outputs "DONE".
B
Use while True with stop_reason as the primary control, with an optional safety cap at a high number (e.g., 50) as a secondary safeguard only.
C
Remove the loop entirely and make separate API calls for each tool execution.
D
Use max_tokens to control when the model stops generating tool calls.
B
CORRECT
Explanation

The primary loop control must be stop_reason. Arbitrary iteration caps as the primary mechanism are an anti-pattern. A very high safety cap is acceptable as a secondary safeguard, but stop_reason drives the actual loop behavior.

3
Subagent Context Isolation
Task 1.2 · Multi-Agent
Your synthesis subagent produces a report that doesn't include findings from the web search subagent, even though the web search completed successfully with relevant results. The coordinator passes the synthesis subagent a prompt saying "Synthesize all findings." What's the root cause?
A
The synthesis agent's system prompt doesn't mention web search findings.
B
Subagents have isolated context — the synthesis agent cannot see the web search agent's results unless the coordinator explicitly passes them in the prompt.
C
The web search agent's results exceed the synthesis agent's context window.
D
The synthesis agent needs the web_search tool in its allowedTools to see web results.
B
CORRECT
Explanation

Subagents have isolated context — they do NOT automatically inherit the coordinator's conversation history or other subagents' results. The coordinator must explicitly include web search findings in the synthesis subagent's prompt. "Synthesize all findings" without the actual data is meaningless to the subagent.

4
Narrow Task Decomposition
Task 1.2 · Coordinator Design
Your multi-agent research system produces comprehensive reports on "impact of AI on healthcare" but when asked about "impact of AI on creative industries," the report only covers visual arts (digital art, graphic design, photography), completely missing music, writing, and film. Each subagent completes successfully. What's the most likely root cause?
A
The web search agent's queries aren't broad enough to find non-visual creative industry sources.
B
The coordinator's task decomposition is too narrow — it only assigned visual arts subtopics to subagents.
C
The synthesis agent filtered out non-visual findings during report generation.
D
The document analysis agent has an overly restrictive relevance filter.
Explanation

When all subagents complete successfully but coverage is missing, the root cause is the coordinator's task decomposition. The subagents worked correctly within their assigned scope — they just weren't assigned the right scope. This is a key exam pattern: narrow decomposition = incomplete coverage.

5
Prerequisite Enforcement
Task 1.4 · Workflow Enforcement
Your support agent processes refunds for customers it hasn't verified 8% of the time, despite system prompt instructions saying "Always verify the customer with get_customer before processing any refund." What's the most effective fix?
A
Add 5 few-shot examples showing correct verification before refund.
B
Make the system prompt instruction BOLD and repeat it 3 times.
C
Add a programmatic prerequisite hook that blocks process_refund until get_customer has returned a verified ID.
D
Remove the process_refund tool and replace it with a tool that requires customer_id as a mandatory parameter.
Explanation

When errors have financial consequences, prompt-based approaches (A, B) have a non-zero failure rate. Programmatic enforcement (C) provides deterministic guarantees. D is a valid approach but adds architectural complexity when a hook cleanly solves the ordering issue.

6
PostToolUse Hook Application
Task 1.5 · Hooks
Your agent receives dates from three different MCP tools: Unix timestamps from the order system, ISO 8601 from the shipping API, and "March 15, 2026" format from the CRM. This inconsistency causes the model to miscompare dates. What's the best approach?
A
Add prompt instructions telling the model to normalize all dates before comparison.
B
Implement a PostToolUse hook that normalizes all date formats to ISO 8601 before the model processes them.
C
Modify all three MCP tools to return dates in ISO 8601 format.
D
Add a separate date normalization tool the model calls after each data retrieval.
Explanation

PostToolUse hooks are designed exactly for this: intercepting tool results for transformation before the model processes them. A is unreliable (probabilistic). C requires changing external systems you may not control. D adds unnecessary round trips and complexity.

7
Parallel Subagent Execution
Task 1.3 · Spawning
Your research pipeline currently processes subagents sequentially: search → analysis → synthesis. The search and analysis could run in parallel since they're independent. What's the correct way to parallelize?
A
Have the coordinator emit multiple Task tool calls in a single response, spawning search and analysis simultaneously.
B
Create a separate coordinator for each parallel branch and merge results afterward.
C
Use Python threading to call the Agent SDK from multiple threads concurrently.
D
Configure subagents to automatically detect when another subagent has completed and start executing.
Explanation

The SDK supports parallel subagent execution by having the coordinator emit multiple Task tool calls in a single response. This is the designed pattern. B adds unnecessary coordination overhead. C works around the SDK instead of using it. D assumes subagent communication that doesn't exist in hub-and-spoke.

8
Hook vs Prompt Enforcement
Task 1.5 · Decision Making
Which of the following should use a prompt instruction rather than a programmatic hook?
A
Blocking refund operations above $500.
B
Requiring customer verification before financial operations.
C
Encouraging the agent to use a friendly, empathetic tone with frustrated customers.
D
Preventing the agent from disclosing internal system information.
Explanation

Tone and style preferences are appropriate for prompt instructions — they don't have critical compliance implications if occasionally imperfect. Financial gates (A, B) and security rules (D) need deterministic enforcement via hooks because failures have concrete consequences.

9
Coordinator Prompt Design
Task 1.3 · Context Passing
When designing a coordinator agent's prompt for delegating to subagents, which approach produces the best results?
A
Step-by-step procedural instructions: "First search for X, then analyze Y, then synthesize Z."
B
Research goals and quality criteria: "Find comprehensive coverage of topic X, ensure findings span subtopics Y and Z, cite all sources."
C
A list of tool names to call in order with fixed parameters.
D
A single-sentence summary: "Research and report on X."
Explanation

Coordinator prompts should specify research goals and quality criteria rather than step-by-step procedures. This enables subagent adaptability — they can adjust their approach based on what they discover. Procedural instructions (A, C) are brittle. Too vague (D) gives no quality guidance.

10
Multi-Concern Request Handling
Task 1.4 · Workflow Design
A customer says: "I want a refund for order #123, also my address needs updating, and can you check the status of order #456?" How should the agent handle this?
A
Handle the first request only and ask the customer to submit separate tickets for each issue.
B
Handle all three sequentially, completing each before starting the next.
C
Decompose into distinct items, investigate each using shared context (customer ID), then synthesize a unified response.
D
Escalate to a human since multi-concern requests are too complex for automation.
Explanation

The agent should decompose multi-concern requests into distinct items, investigate each (potentially in parallel using shared customer context), then synthesize a unified resolution. This is explicitly tested — don't make customers repeat themselves or create separate tickets.

11
Session Resumption
Task 1.7 · Session Management
You investigated a codebase issue yesterday and need to continue today. However, you've pushed several commits since then that changed files the agent analyzed. What's the best approach?
A
Resume the session with --resume and let the agent figure out what changed.
B
Resume the session with --resume and explicitly inform the agent about specific file changes for targeted re-analysis.
C
Start a completely fresh session and re-explore everything from scratch.
D
Use fork_session to create a branch from yesterday's session.
Explanation

When resuming after code changes, inform the agent about specific changes for targeted re-analysis rather than requiring full re-exploration (C) or hoping it notices (A). fork_session (D) is for exploring divergent approaches, not for updated code.

12
fork_session Use Case
Task 1.7 · Forking
You've analyzed a codebase and identified that a function can be refactored using either Strategy A (dependency injection) or Strategy B (factory pattern). Both require different file changes. What's the best way to evaluate both approaches?
A
Implement Strategy A, evaluate it, undo changes with git, then implement Strategy B.
B
Use fork_session to create two independent branches from the shared analysis baseline — one for each strategy — and compare results.
C
Ask Claude to describe both approaches theoretically without implementing either.
D
Create two separate agent instances from scratch and give each one a different strategy.
Explanation

fork_session is designed for creating independent branches from a shared analysis baseline to explore divergent approaches. This preserves the shared codebase understanding while allowing parallel exploration. D loses the shared baseline. A is wasteful. C doesn't test real implementation.

13
Scoped verify_fact Tool
Task 1.2 · Tool Scoping
Your synthesis agent frequently pauses to verify simple facts (dates, names, statistics), requiring round-trips through the coordinator to the web search agent — adding 40% latency. 85% of verifications are simple fact-checks. What's the best optimization?
A
Give the synthesis agent a scoped verify_fact tool for simple lookups, while complex verifications continue through the coordinator.
B
Give the synthesis agent full web search tools to handle all verifications directly.
C
Have the synthesis agent batch all verification needs and return them to the coordinator at once.
D
Pre-cache extra context around each source during initial research.
Explanation

This is the principle of least privilege: give the synthesis agent only what it needs for the 85% common case. B over-provisions (violates separation of concerns). C creates blocking dependencies. D relies on speculative caching that can't predict what needs verification.

When escalating to a human agent, which information is MOST critical to include in the handoff summary?
A
The complete conversation transcript between the AI agent and the customer.
B
The model's internal reasoning chain and confidence scores for each decision.
C
Customer ID, root cause analysis, refund amount, steps already taken, and recommended action.

Code Review Decomposition
Task 1.6 · Task Decomposition
A single-pass code review of a 14-file PR produces detailed feedback for some files but superficial comments for others, with contradictory findings — flagging a pattern as problematic in one file while approving identical code elsewhere. How should you restructure?
A
Split into per-file local analysis passes plus a separate cross-file integration pass examining data flow between files.
B
Require developers to submit smaller PRs of 3-4 files maximum.
C
Switch to a larger context window model to give all 14 files adequate attention.
D
Run three independent reviews on the full PR and flag only issues appearing in 2+ runs.
Explanation

The root cause is attention dilution. Per-file passes ensure consistent depth; a separate integration pass catches cross-file issues. B shifts burden to developers. C misunderstands that larger context ≠ better attention. D suppresses real bug detection by requiring consensus.

Stale Context Decision
Task 1.7 · Session Management
You need to continue an analysis from 3 days ago. The prior session contained extensive tool results from database queries, but the database has been significantly updated since. What's the best approach?
A
Resume the session — the model will re-query if needed.
B
Start a fresh session with an injected summary of key findings, since prior tool results are stale.
C
Resume and use /compact to reduce the stale context.
D
Fork the old session to preserve the original context.

Iterative Refinement Loop
Task 1.2 · Multi-Agent
The coordinator evaluates the synthesis output and finds that the section on "regulatory impact" is thin with only one source cited. What should happen next?
A
Accept the output as-is and note the gap in a disclaimer.
B
Re-run the entire pipeline from scratch with a more detailed initial query.
C
Re-delegate to search and analysis subagents with targeted queries about regulatory impact, then re-invoke synthesis with the expanded findings.

19
Tool Result Appending
Task 1.1 · Implementation
After executing a tool, what's the correct way to return the result to the agentic loop?
A
Start a new API call with only the tool result as the message.
B
Append the assistant's response (with tool_use block) to messages, then append a user message containing the tool_result with the matching tool_use_id.
C
Modify the original user message to include the tool result.
D
Append the tool result directly as an assistant message.
Explanation

The correct pattern is: (1) append the assistant's response containing the tool_use block, then (2) append a user message with the tool_result referencing the tool_use_id. This preserves the conversation structure so the model can reason about the result in the next iteration.

20
Adaptive vs Fixed Decomposition
Task 1.6 · Strategy Selection
You need to add comprehensive tests to a large legacy codebase with no existing test infrastructure. Which decomposition strategy is most appropriate?
A
Fixed sequential: analyze each file → write tests → run tests → repeat for every file.
B
Dynamic adaptive: map codebase structure → identify high-impact areas → create a prioritized test plan that adapts as dependencies are discovered.
C
Parallel: assign each directory to a separate agent working simultaneously.
D
Single pass: analyze the entire codebase at once and generate all tests in one shot.
Explanation

An open-ended task with unknown scope ("add comprehensive tests to a legacy codebase") requires dynamic adaptive decomposition. You don't know what you'll find until you start exploring. Fixed sequential (A) assumes you know the structure. Parallel (C) risks duplicate work. Single pass (D) causes attention dilution at this scale.
