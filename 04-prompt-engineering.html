Core Concepts

Six Critical Concepts

1. tool_use + JSON Schemas = Guaranteed Structure

Using tool_use with JSON schemas is the most reliable approach for guaranteed schema-compliant output. It eliminates JSON syntax errors entirely. BUT: it does NOT prevent semantic errors (line items that don't sum to total, values in wrong fields).

Python — Schema Design Best Practices
extract_tool = {
    "name": "extract_invoice",
    "input_schema": {
        "type": "object",
        "properties": {
            // ✅ Required fields — must exist in source
            "vendor_name": {"type": "string"},
            "total": {"type": "number"},

            // ✅ NULLABLE fields — prevents hallucination
            // If info isn't in doc, model returns null instead of fabricating
            "invoice_number": {"type": ["string", "null"]},
            "due_date": {"type": ["string", "null"]},

            // ✅ Enum with "other" + detail pattern
            "category": {"type": "string",
                "enum": ["supplies", "services", "equipment", "other"]},
            "category_detail": {"type": ["string", "null"]},

            // ✅ Self-check fields for semantic validation
            "calculated_total": {"type": "number"},
            "conflict_detected": {"type": "boolean"},

            // ✅ Confidence for human review routing
            "confidence": {"type": "string",
                "enum": ["high", "medium", "low", "unclear"]}
        },
        "required": ["vendor_name", "total", "confidence"]
    }
}

3. Few-Shot Prompting — When & Why

Few-shot examples are the most effective technique for achieving consistently formatted, actionable output when detailed instructions alone produce inconsistent results. They enable generalization to novel patterns (not just matching pre-specified cases). Use 2-4 targeted examples for ambiguous scenarios showing reasoning for why one action was chosen over alternatives.

Few-Shot vs Explicit Criteria
Few-shot examples show how to handle ambiguity. Explicit criteria define what to look for. Both are needed — explicit criteria for precision, few-shot for consistency. Neither alone is sufficient for complex extraction tasks.
4. Validation-Retry Loop — When Retries Work vs Don't

✅ Retries WILL Help

Format mismatches (wrong date format), structural errors (missing required field), JSON parsing issues, extraction from wrong section of document

❌ Retries WON'T Help

Information simply absent from source document, data only exists in external document not provided, ambiguous source data with no clear answer

Semantic vs Syntax Errors
Task 4.3 · Limitations
Your extraction uses tool_use with JSON schemas. The output is valid JSON matching the schema, but line item amounts don't sum to the total. What type of error is this?
A
Schema validation error — the schema needs stricter constraints.
B
Semantic error — tool_use eliminates syntax errors but NOT semantic errors like incorrect calculations.
C
Model hallucination — the model is fabricating numbers.
D
Context window overflow — the model lost track of earlier line items.
Explanation

Tool_use guarantees syntax compliance but NOT semantic correctness. Values in wrong fields, incorrect sums, and logical inconsistencies are semantic errors that require separate validation (e.g., comparing calculated_total vs stated_total).

3
Nullable Fields to Prevent Hallucination
Task 4.3 · Schema Design
Your extraction schema marks invoice_number as required, but some documents don't contain an invoice number. The model fabricates plausible-looking invoice numbers. What's the fix?
A
Add "only extract information that actually appears in the document" to the prompt.
B
Make the field nullable: {"type": ["string", "null"]} — the model returns null when info isn't present instead of fabricating.
C
Remove the field entirely from the schema.
D
Add a confidence score and filter low-confidence values.
Explanation

When source documents may not contain certain information, make those fields optional (nullable). Required fields force the model to fill them — even by fabricating. Nullable fields give the model permission to return null.

Validation-Retry Effectiveness
Task 4.4 · Retry Loops
Extraction fails because the source document mentions "see appendix for pricing details" but the appendix wasn't included. Retry with error feedback continues to fail. Why?
A
The retry prompt isn't specific enough about what went wrong.
B
The required information exists only in an external document not provided — retries are ineffective when data is absent from the source.
C
The model needs more context window to process the document.
D
The schema should make the pricing field optional.
Explanation

Retries work for format/structural errors. They are ineffective when information is simply absent from the source. No amount of retrying will produce data that doesn't exist in the provided document. D is a valid schema improvement but doesn't address the retry question.

False Positive Reduction
Task 4.1 · Explicit Criteria
Your code review generates many false positives in the "comment accuracy" category, undermining developer trust even in categories where it's accurate. What's the best first step?
A
Add "be more conservative" to the review prompt.
B
Temporarily disable the high-false-positive category while improving its prompts with explicit criteria.
C

"Be conservative" is vague and fails to improve precision. Temporarily disabling high-false-positive categories restores developer trust while you improve prompts. Trust is critical — if developers dismiss all findings, accurate categories are wasted too.

tool_choice: "any" Use Case
Task 4.3 · Configuration
You have three extraction schemas (invoice, receipt, purchase order) but don't know the document type in advance. The model sometimes returns text instead of calling any extraction tool. What tool_choice setting fixes this?
A
tool_choice: "auto" — the model can figure out the right one.
B
tool_choice: "any" — guarantees the model calls a tool but lets it choose which extraction schema.
C
Force the invoice tool and handle mismatches in validation.
D

Explanation

"any" guarantees the model calls a tool (no text response) but lets it choose which one. Perfect when you have multiple valid tools and need structured output guaranteed. "auto" (A) allows text responses. Forced (C) assumes you know the type.

Explicit Criteria vs Vague Instructions
Task 4.1 · Precision
Your review prompt says "only report high-confidence findings." The false positive rate hasn't improved. Why?
A
General instructions like "be conservative" or "high-confidence only" fail to improve precision — specific categorical criteria defining WHAT to report vs skip are needed.
B
The model's confidence calibration needs fine-tuning.
C
The model is ignoring the instruction due to other conflicting prompts.
D
Extended thinking is needed for the model to properly assess confidence.
Explanation

Vague instructions ("be conservative", "high-confidence only") don't give the model actionable criteria. Instead: specific categorical criteria — "report bugs and security issues, skip minor style and local patterns" with concrete code examples for each severity level.

Enum + Other Pattern
Task 4.3 · Schema Design
Your category field has "enum": ["hardware", "software", "networking"]. Documents occasionally describe categories that don't fit any option, causing misclassification. What's the schema fix?
A
Remove the enum and use a free-text string field.
B
Add "other" to the enum and a companion category_detail nullable string field for description.
C
Add more enum values to cover all possible categories.
D
Let the model choose the closest matching enum value.
Explanation

The "other" + detail pattern: add "other" to the enum for extensible categorization, with a companion detail field for description. This gives structure while handling unexpected categories. A loses all structure. C can't anticipate everything. D forces inaccurate classification.

Prompt Refinement Before Batch
Task 4.5 · Cost Optimization
You need to batch-process 10,000 documents. What should you do before submitting the full batch?
A
Submit all 10,000 immediately and fix issues in a second pass.
B
Run prompt refinement on a sample set (e.g., 50-100 documents) to maximize first-pass success rates before processing the full volume.
C
Split into batches of 1,000 and process sequentially.
D
Validate the JSON schema against all 10,000 documents first.
Explanation

Prompt refinement on a sample set before large-volume batch processing maximizes first-pass success rates and reduces resubmission costs. Testing with 10,000 immediately wastes API credits on a potentially flawed prompt.

Independent Review Instance
Task 4.6 · Multi-Instance
Your pipeline generates code with Claude, then asks the SAME Claude session to review it. The review finds very few issues. What architectural change would catch more bugs?
A
Add "be extra critical" to the review prompt.
B
Use a second independent Claude instance (without the generator's reasoning context) for review.
C
Use extended thinking for the review pass.
D
Have the model generate 3 versions and review each against the others.
Explanation

Self-review is limited because the model retains reasoning context from generation. An independent review instance without that context is more effective at questioning assumptions and catching subtle issues.
