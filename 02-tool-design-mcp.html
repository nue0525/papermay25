Core Concepts

The Five Pillars of Domain 2

This domain tests whether you can design tools that LLMs can reliably select, implement proper error handling for MCP tools, and configure MCP servers for team workflows.

Tool Descriptions = Tool Selection

Tool descriptions are the primary mechanism LLMs use to decide which tool to call. Minimal descriptions → unreliable selection → wrong tool calls. Descriptions must include: purpose, input formats, example queries, edge cases, boundaries vs similar tools.

Structured Error Responses

MCP tools must return structured errors with errorCategory (transient/validation/permission/business), isRetryable boolean, and human-readable descriptions. Generic "Operation failed" prevents intelligent recovery.

Tool Distribution & Scoping

Giving an agent 18+ tools degrades selection reliability. Each subagent should get only tools for its role. Use tool_choice config: "auto" (may return text), "any" (must call a tool), forced (must call specific tool).

Built-in Tool Selection

Grep = search file contents (patterns, functions, errors). Glob = find files by path/name pattern. Read = full file content. Edit = targeted modification (unique text match). Use Read+Write when Edit fails on non-unique text.

Good vs Bad Tool Descriptions

❌ BAD — Minimal Description
{
  "name": "get_customer",
  "description": "Retrieves customer information"
  // ❌ What information? What input?
  // ❌ When to use vs lookup_order?
  // ❌ Model can't differentiate
}

{
  "name": "lookup_order",
  "description": "Retrieves order details"
  // ❌ Nearly identical to above
  // ❌ Both accept similar identifiers
}
✅ GOOD — Differentiated Description
{
  "name": "get_customer",
  "description": """Retrieves verified
customer profile by ID or email.
Returns: name, email, account_status,
verification_level, address.
Use BEFORE any order or refund ops.
Do NOT use for order lookups."""
}
{
  "name": "lookup_order",
  "description": """Retrieves order details
by order ID (format: ORD-XXXXX).
Returns: items, total, status, shipping.
For order-specific queries only.
Requires verified customer_id first."""
}

Shared via version control. All team members get these tools.

// ✅ No secrets in code

Personal/experimental. NOT shared with team.

MCP Structured Error Response Pattern

Python — Complete Error Response Pattern
# ✅ TRANSIENT error — retryable
{
    "isError": True,
    "errorCategory": "transient",       # timeout, service down
    "isRetryable": True,
    "description": "Database connection timed out after 5s",
    "attempted": "SELECT * FROM orders WHERE id = 'ORD-123'"
}

# ✅ VALIDATION error — NOT retryable without fix
{
    "isError": True,
    "errorCategory": "validation",     # invalid input
    "isRetryable": False,
    "description": "Order ID must be format ORD-XXXXX, got: '12345'"
}

# ✅ BUSINESS error — customer-facing explanation needed
{
    "isError": True,
    "errorCategory": "business",       # policy violation
    "isRetryable": False,
    "description": "Refund window (30 days) has expired for this order",
    "customerMessage": "This order is past the 30-day refund period."
}

# ✅ PERMISSION error — NOT retryable
{
    "isError": True,
    "errorCategory": "permission",
    "isRetryable": False,
    "description": "Agent lacks access to premium tier orders"
}

# ✅ VALID EMPTY RESULT — this is NOT an error!
{
    "found": False,
    "message": "No orders found matching criteria"
    # No isError flag — this is a successful query with no matches
}
Critical Distinction
Access failures (timeouts, service unavailable) need retry decisions. Valid empty results (successful query, no matches) are NOT errors. Confusing these causes agents to retry queries that correctly returned zero results, or to accept failures as "no data found."

Tool Description as First Fix
Task 2.1 · Tool Interfaces
Your agent calls analyze_content when it should call analyze_document. Both have near-identical descriptions. What's the most effective first step?
A
Add few-shot examples showing correct routing in the system prompt.
B
Rename tools and expand descriptions to clearly differentiate each tool's purpose, inputs, outputs, and boundaries.
C
Implement a routing classifier to pre-select the appropriate tool.
D
Consolidate both into a single analyze tool.
Explanation

Tool descriptions are the primary mechanism for tool selection. Fix the root cause first (descriptions), not symptoms. Rename analyze_content to extract_web_results with a web-specific description. A adds token overhead without fixing the underlying issue.

Error Category Selection
Task 2.2 · Error Responses
Your MCP tool queries a database and gets a connection timeout. The tool returns {"isError": true, "message": "Error occurred"}. The agent doesn't retry and tells the user "no data was found." What's wrong?
A
The tool needs to implement automatic retry logic internally.
B
The error response lacks errorCategory: "transient" and isRetryable: true — the agent can't distinguish a timeout from a valid empty result.
C
The agent's system prompt needs instructions to retry on errors.
D
The tool should throw an exception instead of returning an error object.
Explanation

Without structured error metadata (errorCategory, isRetryable), the agent can't make intelligent recovery decisions. A generic "Error occurred" message is indistinguishable from "no results found" to the agent. The tool must communicate what type of error occurred and whether retry is appropriate.

Tool Count and Selection
Task 2.3 · Tool Distribution
Your single agent has access to 18 tools. It frequently selects the wrong tool for similar operations — e.g., calling update_billing when it should call update_shipping. What's the primary issue?
A
Too many tools increases decision complexity, degrading tool selection reliability. Restrict each agent to 4-5 tools relevant to its role.
B
The model needs a larger context window to process all 18 tool descriptions.
C
The tools need to be consolidated into fewer, more general tools.
D
Adding more few-shot examples for each tool will resolve the confusion.
Explanation

Having 18 tools increases decision complexity — the model must differentiate between many similar options. The fix is to split into specialized agents with scoped tool access (4-5 each). C risks losing specificity. D adds more tokens without reducing complexity.

tool_choice Configuration
Task 2.3 · Tool Choice
You need the model to extract metadata from a document before any enrichment tools run. The model sometimes skips the extraction step and jumps to enrichment. What's the correct tool_choice configuration?
A
tool_choice: "auto" with stronger prompt instructions.
B
tool_choice: "any" to guarantee a tool is called.
C
tool_choice: {"type": "tool", "name": "extract_metadata"} to force the specific extraction tool first.
D
Remove all enrichment tools from the first API call and add them in a subsequent call.
Explanation

Forced tool selection ensures a specific tool is called first. Then in the follow-up turn (with the extraction result in context), you can switch to "auto" for enrichment. "any" (B) guarantees A tool but not WHICH tool. D works but is less clean than forced selection.

Environment Variable Expansion
Task 2.4 · Secrets Management
Your team needs to share MCP server configuration via version control, but the server requires an API token. What's the correct approach?
A
Add the token directly in .mcp.json and add the file to .gitignore.
B
Use environment variable expansion in .mcp.json: "API_TOKEN": "${MY_API_TOKEN}"
C
Store the token in CLAUDE.md which the agent can read at runtime.
D
Create a separate secrets.json file referenced by .mcp.json.
Explanation

.mcp.json supports environment variable expansion using ${VAR_NAME} syntax. This lets you commit the config structure to version control while keeping secrets in each developer's environment. A commits secrets or removes the config from version control.

Grep vs Glob Selection
Task 2.5 · Built-in Tools
You need to find all files in the project that call the processPayment() function. Which built-in tool should you use?
A
Grep — searches file contents for the function name pattern.
B
Glob — finds files matching a naming pattern.
C
Read — load each file and search manually.
D
Bash — run a shell find command.
Explanation

Grep searches file contents for patterns. Glob finds files by name/path pattern. You want to find what's inside files, not find files by their names. This is a frequently tested distinction.

8
Finding Test Files
Task 2.5 · Built-in Tools
You need to find all test files in the project (named *.test.tsx or *.spec.ts). Which tool?
A
Grep — search for "test" or "spec" in file contents.
B
Glob — find files matching **/*.test.tsx and **/*.spec.ts patterns.
C
Bash — run ls -R | grep test.
D
Read — start from the root and recursively read directories.
Explanation

This is a file path pattern match — exactly what Glob is for. Grep searches file contents, which is wrong here. You want files BY NAME, not files that contain certain text.

Edit vs Read+Write Fallback
Task 2.5 · File Operations
The Edit tool fails with "non-unique text match" when trying to modify a function. What's the correct fallback?
A
Add more surrounding context to the Edit anchor text to make it unique.
B
Use Bash with sed to perform the replacement.
C
Use Read to load the full file, modify the content, then Write the complete file back.
D
Delete the file and recreate it with the modifications.
Explanation

When Edit fails due to non-unique text matches, Read + Write is the documented fallback. Read the full file, modify in memory, Write back. A might work but isn't the standard fallback pattern. B uses shell tools unnecessarily. D is destructive.

Add an operation_type enum parameter to the tool.

Empty Results vs Access Failures
Task 2.2 · Error Handling
An agent keeps retrying a customer lookup that consistently returns empty results. The tool returns the same response for "customer not found" and "database timeout." What's the root issue?
A
The agent needs a maximum retry limit in its system prompt.
B
The tool doesn't distinguish between access failures (errors needing retry decisions) and valid empty results (successful queries with no matches).
C
The agent should only retry if the user explicitly asks to try again.
D
The database needs a health check endpoint the tool queries first.
Explanation

This is a critical distinction on the exam. Access failures (timeout, service down) may be worth retrying. Valid empty results (query worked, no matches) should NOT be retried. The tool must clearly differentiate these in its response structure.

Incremental Codebase Exploration
Task 2.5 · Exploration Pattern
You need to understand how the payment processing module works in an unfamiliar codebase. What's the recommended approach?
A
Use Read on every file in the payments directory to understand the full module.
B
Start with Grep to find entry points (e.g., "processPayment"), then Read to follow imports and trace flows incrementally.
