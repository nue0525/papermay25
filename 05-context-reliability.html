Core Concepts

Six Critical Context Patterns

1. Progressive Summarization Risk

Condensing numerical values, percentages, dates, and customer-stated expectations into vague summaries loses critical data. Extract transactional facts (amounts, dates, order numbers) into a persistent "case facts" block outside summarized history.

2. "Lost in the Middle" Effect

Models reliably process information at the beginning and end of long inputs but may omit findings from middle sections. Fix: place key findings summaries at beginning, use explicit section headers, organize results to mitigate position effects.

3. Escalation Triggers (3 Valid)

Always escalate: (1) Customer explicitly requests human, (2) Policy is ambiguous or silent on the request, (3) Agent cannot make meaningful progress. Never use: sentiment-based escalation or self-reported confidence scores — both are unreliable proxies.

4. Error Propagation (Not Suppression)

Two anti-patterns: (1) silently suppress errors (returning empty results as success), (2) terminate entire workflow on single failure. Correct: return structured error context (failure type, attempted query, partial results, alternatives) to enable coordinator recovery.

5. Scratchpad Files for Long Sessions

In extended sessions, models start giving inconsistent answers referencing "typical patterns" instead of specific classes discovered earlier. Have agents maintain scratchpad files recording key findings, then reference them for subsequent questions.

6. Information Provenance

Source attribution is lost during summarization when findings are compressed without preserving claim-source mappings. Require subagents to output structured claim-source mappings (URLs, document names, excerpts) that downstream agents preserve through synthesis.

Key Anti-Patterns for Exam
Trim verbose tool outputs BEFORE they accumulate. An order lookup returning 40+ fields when only 5 are relevant consumes tokens disproportionately. PostToolUse hooks can trim to relevant fields. Also: requiring subagents to include metadata (dates, source locations) in structured outputs supports accurate downstream synthesis.

Context Trimming Pattern

Python — Trim Verbose Tool Output
# ✅ BEFORE: Order lookup returns 40+ fields
raw_order = {
    "order_id": "ORD-123", "status": "shipped",
    "items": [...], "total": 150.00,
    "internal_tracking_id": "...",  # irrelevant
    "warehouse_code": "...",       # irrelevant
    "fulfillment_batch": "...",     # irrelevant
    # ... 35 more internal fields
}

# ✅ AFTER: PostToolUse hook trims to relevant fields
class TrimOrderHook:
    RELEVANT = ["order_id","status","items","total","created_at","return_eligible"]

    def post_tool_use(self, tool_name, tool_input, result):
        if tool_name == "lookup_order":
            return {k: v for k, v in result.items() if k in self.RELEVANT}
        return result

# ✅ Persistent "case facts" block for multi-issue sessions
case_facts = {
    "customer_id": "CUST-456",
    "issues": [
        {"order": "ORD-123", "amount": 150.00, "status": "pending_refund"},
        {"order": "ORD-456", "amount": 75.00, "status": "investigating"}
    ]
}
# Include case_facts in EVERY prompt, outside summarized history

1
Progressive Summarization
Task 5.1 · Context Preservation
After a long support conversation, the agent tells the customer their refund amount is "approximately $150" when the actual amount was $147.83. The agent had the correct amount earlier. What went wrong?
A
Progressive summarization condensed the exact amount into a vague approximation. Exact values should be extracted into a persistent "case facts" block.
B
The context window overflowed and the exact amount was truncated.
C
The model rounded for brevity in its response to the customer.
D
The tool returned an approximate value in a subsequent lookup.
Explanation

Progressive summarization loses numerical precision. $147.83 becomes "approximately $150." Fix: extract transactional facts (exact amounts, dates, order numbers) into a persistent "case facts" block included in each prompt, outside the summarized conversation history.

Lost in the Middle

A
The middle sources were lower quality and correctly deprioritized.
B
"Lost in the middle" effect — place key findings summaries at the beginning and organize with explicit section headers.

D
The synthesis agent has a bias toward primary sources listed first.
Explanation

Models reliably process beginning and end of long inputs but may miss middle sections. Fix: place key findings at the beginning, use explicit section headers, and organize results to mitigate position effects.

Escalation: Customer Request
Task 5.2 · Escalation
A customer says "I just want to talk to a real person please." The agent has a simple solution to their problem. What should the agent do?
A
Resolve the issue first since it's simple, then offer the human if still needed.
B
Escalate to a human agent immediately — honor explicit customer requests without attempting resolution first.
C
Acknowledge the request and offer to resolve before transferring.
D
Ask why they want a human to determine if escalation is truly necessary.
Explanation

When a customer explicitly requests a human, escalate immediately. Do not attempt resolution first — this frustrates the customer. This is an absolute escalation trigger.

4
Sentiment vs Complexity
Task 5.2 · Escalation
A customer is angry and uses profanity but their actual request is a simple order status check. Should the agent escalate?
A
Yes — high negative sentiment triggers automatic escalation.
B
No — acknowledge frustration while resolving the straightforward request. Escalate only if the customer explicitly asks for a human.
C
Yes — profanity indicates the customer needs human empathy.
D
No — ignore the sentiment entirely and just provide the order status.
Explanation

Sentiment does not correlate with case complexity. The issue is simple. Acknowledge frustration (don't ignore it — D is wrong) while offering resolution. Sentiment-based escalation is explicitly called out as an unreliable proxy.

Error Propagation Design
Task 5.3 · Multi-Agent
A web search subagent times out. It returns {"status": "search unavailable"} to the coordinator. The coordinator can't determine whether to retry, use alternatives, or proceed with partial results. What's wrong?
A
The error response hides valuable context. Return structured error with failure type, attempted query, partial results, and alternative approaches.
B
The coordinator should have automatic retry logic for all failures.
C
The subagent should keep retrying internally until it succeeds.
D
The coordinator should terminate the workflow since search is unavailable.
Explanation

Generic error messages like "search unavailable" hide valuable context from the coordinator. Return structured error context so the coordinator can make informed recovery decisions. D is an anti-pattern (terminating entire workflow on single failure).

6
Context Degradation in Long Sessions
Task 5.4 · Codebase Exploration
After 2 hours of codebase exploration, your agent starts referencing "typical patterns" and giving inconsistent answers about classes it analyzed earlier. What's the recommended approach?
A
Use /compact to reduce context and continue.
B
Have the agent maintain scratchpad files recording key findings, referencing them for subsequent questions to counteract degradation.
C
Start a completely fresh session.
D
Increase the model's context window size.
Explanation

Context degradation is a known issue in extended sessions. Scratchpad files persist key findings across context boundaries, acting as external memory the agent can reference. /compact (A) helps but may lose important details. Fresh session (C) loses all progress.

Human Review Sampling
Task 5.5 · Human Review
Your extraction system has 97% overall accuracy. Management wants to reduce human review. What should you verify before automating?
A
97% is high enough — automate all high-confidence extractions immediately.
B
Analyze accuracy by document type and field — aggregate 97% may mask poor performance on specific document types or fields.
C
Run a single validation pass on 100 random documents.
D
Check if the model's self-reported confidence correlates with actual accuracy.
Explanation

Aggregate metrics can mask poor performance on specific segments. 97% overall might mean 99.5% on invoices but 80% on handwritten receipts. Analyze by document type AND field before reducing review.

Conflicting Sources in Synthesis
Task 5.6 · Provenance
Two credible sources report different statistics for the same metric ("30% increase" vs "45% increase"). How should the synthesis output handle this?
A
Average the two values and report 37.5%.
B
Use the more recent source.
C
Include both values with source attribution, annotating the conflict rather than arbitrarily selecting one.
D
Exclude both since the data is unreliable.
Explanation

When credible sources conflict, preserve both with source attribution. Let the reader see the conflict. Don't average, don't arbitrarily pick one. Structure the report to distinguish well-established from contested findings.

9
Verbose Tool Output Accumulation
Task 5.1 · Token Management
Your customer support agent makes 6 order lookups during a multi-issue session. Each returns 40+ fields. The context fills up rapidly. What's the best prevention?
A
Trim verbose tool outputs to only relevant fields before they accumulate in context (e.g., keep only 5 return-relevant fields from order lookups).
B
Summarize previous tool results at each step.
C
Use a larger context window model.
D
Limit the agent to 3 tool calls per session.
Explanation

Tool results consume tokens disproportionately to their relevance. Trim to relevant fields via PostToolUse hooks BEFORE they accumulate. Prevention is better than cure — don't wait until context is full.

Multiple Customer Matches
Task 5.2 · Ambiguity Resolution
A customer says "I'm John Smith." The get_customer tool returns 3 matches for "John Smith." What should the agent do?
A
Select the most recently active account.
B
Select the account with the most orders.
C
Ask the customer for additional identifiers (email, order number, phone) to disambiguate.
D
Escalate to a human since the system can't determine which account.
Explanation

When multiple matches exist, ask for additional identifiers rather than selecting based on heuristics (A, B). Heuristic selection risks acting on the wrong account — with financial consequences.

Policy Gap Escalation
Task 5.2 · Escalation
A customer asks for a price match against a competitor's offer. Your company's return policy covers own-site price adjustments but doesn't mention competitor matching. What should the agent do?
A
Deny the request since the policy doesn't explicitly allow competitor matching.
B
Escalate to a human — the policy is ambiguous/silent on this specific request, which is an escalation trigger.

Ask the customer if they'd accept a discount code instead.

When policy is ambiguous or silent on the customer's specific request, this is an explicit escalation trigger. The agent shouldn't guess at policy (A denies without authority, C approves without authority).

Source Attribution in Synthesis
Task 5.6 · Provenance
Your synthesis agent produces a report where claims can't be traced back to sources. The web search agent's findings were compressed during summarization. How do you prevent this?
A
Require subagents to output structured claim-source mappings (source URLs, document names, excerpts) that downstream agents preserve through synthesis.
B
Include all raw source material in the synthesis prompt.
C
Add "cite your sources" to the synthesis prompt.
D
Have the synthesis agent re-search for sources to verify claims.
Explanation

Source attribution is lost during summarization. The fix: require subagents to output structured claim-source mappings that the synthesis agent must preserve. This is architectural, not prompt-based (C is insufficient).

13
Crash Recovery
Task 5.4 · State Persistence
Your multi-agent research pipeline crashes mid-execution. You want to resume from where it left off without re-doing completed work. What's the best architecture?
A
Re-run the entire pipeline with caching for completed steps.
B
Design structured state persistence: each agent exports state to a known location, and the coordinator loads a manifest on resume and injects into agent prompts.
C
Use session resumption with --resume to continue from the last point.
D
Log all API calls and replay them on restart.
Explanation

Structured state persistence via agent state exports (manifests) that the coordinator loads on resume is the robust pattern. C assumes a single Claude Code session. D is brittle and may produce different results on replay.

14
Temporal Data in Sources
Task 5.6 · Provenance
Two sources report different market share numbers. On investigation, one is from 2024 and the other from 2026 — they're not contradictory, just different time periods. How should you prevent this misinterpretation?
A
Use only the most recent source.
B
Require subagents to include publication or data collection dates in structured outputs to enable correct temporal interpretation.
C
Add "check dates before comparing" to the synthesis prompt.
D
Exclude sources older than 1 year.
Explanation

Without dates, temporal differences are misinterpreted as contradictions. Requiring structured outputs with publication/collection dates is architectural — the data is always available for correct interpretation. C is prompt-based and less reliable.
