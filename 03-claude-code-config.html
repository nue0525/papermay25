# Codebase Analysis Skill

Analyze the provided directory structure, identify key patterns,
dependencies, and architectural decisions. Return a structured summary.
Key Distinction: Skills vs CLAUDE.md vs Rules
CLAUDE.md = always-loaded universal standards. Skills = on-demand invocation for specific tasks. .claude/rules/ = conditionally loaded based on file glob patterns. Know when to use each!

CI/CD Integration — Critical Flags

Bash — CI/CD Pipeline Commands
# ✅ CORRECT: Non-interactive mode for CI
claude -p "Review this PR for security issues"

# ✅ Structured JSON output for automated processing
claude -p "Review this code" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"findings":{...}}}'

# ❌ DOES NOT EXIST — exam distractors:
# CLAUDE_HEADLESS=true   ← NOT a real env var
# claude --batch         ← NOT a real flag
# claude < /dev/null     ← Unix workaround, not proper

New Team Member Missing Config
Task 3.1 · Hierarchy
A new team member doesn't get the coding standards that the team lead has. The lead's configuration works perfectly. Where is the config most likely placed incorrectly?
A
In the project's .claude/CLAUDE.md
B
In the team lead's ~/.claude/CLAUDE.md (user-level, not shared)
C
In .claude/rules/ with incorrect glob patterns
D

User-level ~/.claude/CLAUDE.md applies only to that user — NOT shared via version control. The fix: move to project-level .claude/CLAUDE.md.

Path-Specific Rules vs Directory CLAUDE.md
Task 3.3 · Rules
Your testing conventions must apply to all test files (*.test.tsx) regardless of which directory they're in. Tests are spread throughout the codebase next to the code they test. What's the most maintainable approach?
A
Place a CLAUDE.md with testing conventions in every directory that contains test files.
B
Create a .claude/rules/testing.md with paths: ["**/*.test.tsx"] in the YAML frontmatter.
C
Put all testing conventions in the root CLAUDE.md and rely on Claude to infer when they apply.
D
Create a testing skill that developers invoke before writing tests.
Explanation

Glob-pattern rules in .claude/rules/ with YAML frontmatter paths load only when editing matching files, regardless of directory. A is unmaintainable with test files everywhere. C relies on inference. D requires manual invocation.

Direct execution with comprehensive upfront instructions detailing every service.

CI Pipeline Hang
Task 3.6 · CI/CD
Your CI job runs claude "Analyze this PR" but hangs indefinitely. Logs show Claude Code waiting for interactive input. What's the fix?
A
Use the -p flag: claude -p "Analyze this PR"
B
Set CLAUDE_HEADLESS=true
C
Redirect stdin: claude "..." < /dev/null
D
Add --batch flag
Explanation

-p / --print is THE documented flag for non-interactive CI mode. CLAUDE_HEADLESS and --batch DO NOT EXIST — they are exam distractors. < /dev/null is a Unix workaround, not proper.

Test-Driven Iteration
Task 3.5 · Refinement
You want Claude to build a migration script that handles edge cases correctly. What's the most reliable iterative approach?
A
Describe all edge cases verbally and have Claude implement them all at once.
B
Write a test suite with edge cases first (null values, empty strings, special characters), then have Claude iterate by sharing test failures.
C
Have Claude generate the script, then manually test each edge case.
D
Provide few-shot examples of correct edge case handling.
Explanation

Test-driven iteration: write tests first covering expected behavior, edge cases, and performance requirements. Then iterate by sharing test failures to guide progressive improvement. The test failures give Claude precise, actionable feedback.

Your CI system needs to post Claude's review findings as inline PR comments. The findings need to be machine-parseable with file path, line number, and severity. How do you get structured output?
A

B
Use --output-format json with --json-schema to enforce structured output.

14
Self-Review Limitation
Task 3.6 · Review Architecture
Your CI pipeline uses the same Claude session that generated code to also review it. The review rarely finds issues. Why?
A
The model retains reasoning context from generation, making it less likely to question its own decisions. An independent review instance is more effective.
B
The review prompt isn't detailed enough.
C
The code was generated correctly and there are genuinely no issues.
D
The model needs more context window to hold both the code and review criteria.
Explanation

Self-review limitation: the model retains reasoning context from generation. It's anchored to its own decisions. An independent Claude instance without the generator's reasoning context is significantly more effective at catching subtle issues.

15
Explore Subagent
Task 3.4 · Plan Mode
During a multi-phase investigation, the main agent's context fills with verbose discovery output from exploring a large codebase. What's the best approach?
A
Run /compact after each exploration phase.
B
Use the Explore subagent for verbose discovery phases — it isolates output and returns summaries to preserve main conversation context.
C
Start a new session for each exploration phase.
D
Limit the number of files the agent can read per phase.
Explanation

The Explore subagent isolates verbose discovery output while returning summaries to the main agent. This prevents context window exhaustion during multi-phase tasks. /compact (A) compresses but may lose important details.
